# 单品深度调研报告生成器（信息密度版 v4）

针对**某一款**科技新品，生成**高信息密度、可快速扫读**的深度调研报告，输出为简洁克制的 HTML 页面，嵌入真实发布数据、官方高清图与数据可视化。

> **v4 与旧版的根本区别**：旧版追求「留白 > 装饰」，结果是信息稀疏、滚动很久才看到一个数据点。v4 反过来——**密度优先**：固定 8 段结构、参数用 diff 箭头呈现变化、关键洞察前置、全量参数折叠收纳。目标是「一屏看到一堆有效信息」，而不是「一屏一句话」。

## 核心理念（v4 必读）

**信息密度 > 留白，但克制 > 杂乱。**

- 每一屏都要有可读取的有效信息（数字、变化、判断），不要为了"高级感"留大片空白。
- 但密度不等于堆砌：用网格、表格、折叠把信息**组织**得整齐，而不是糊成一团。
- 报告的灵魂是**"变化"和"判断"**：这代升级了什么？哪些是真升级、哪些是参数通胀？值不值？把这些讲清楚，比罗列规格重要。

## 执行流程

```
1. 明确调研对象  → 确认产品、品类、参照对象（上代 / 竞品）
2. 数据收集      → web_search 搜真实发布信息 + 官方/可信媒体高清图
3. 套用固定结构  → 8 段式（见第三步），按品类自适应参数维度
4. 生成报告      → 输出高密度 HTML + 官方图 + Chart.js
5. 预览与询问    → present_files 交付，问用户是否调整
```

## 第一步：明确调研对象

从对话中提取或向用户确认（已说明的不要重复问）：
- **哪一款产品？**（精确到型号）
- **什么品类？**（手机 / 芯片 / 云台相机 / 智能穿戴 / 平板…，决定参数维度）
- **参照对象？**——v4 的对照是核心，确认两条对照轴：
  - **纵向**：上代 / 同系列前作（用于「与上代变化」段）
  - **横向**：1–2 个主要竞品（用于「竞品 SWOT」段）
  - 若是品类首作（无上代），纵向轴改用「同品类标杆」充当参照，并在报告中注明「首作，无上代」。

## 第二步：数据收集

**必须用 web_search 搜真实发布信息，绝不编造数字。** 至少搜索：

```
搜索1: "[产品全名] 发布 价格 配置 [年份]"
搜索2: "[产品全名] 官方 规格参数 [年份]"   → 优先抓官方规格页，以官方为准
搜索3: "[产品全名] 评测 亮点 上手 [年份]"   → 取媒体上手反馈、争议点
搜索4: "[产品全名] vs [上代/竞品] 对比 升级"  → 喂给"变化"与"SWOT"两段
```

收集的数据类型：发布信息、完整规格、配色与定价矩阵、核心升级点、与上代/竞品的参数差、媒体评测与争议点。

**冲突处理**：官方规格页与爆料/媒体数据冲突时，**以官方页为准**，并可在注释里一句话点明差异（这本身就是有信息量的内容）。

### 价格统一规则（强制，沿用）

- **所有价格一律人民币「¥」**，不混用 $ / €。
- 外币信源按发布时点汇率换算，数字后小字注明 `（约合，原价 $799）`。
- 国行已有官方定价的，以官方人民币定价为准，不再换算。
- 不确定的标「待官方公布」，不硬填。

### 官方高清图片获取规则（强制，沿用）

优先级：① 品牌官网产品页 → ② 官方新闻稿 / newsroom → ③ 可信科技媒体发布报道首图（IT之家、少数派、Engadget、DPReview、The Verge 等）。

**获取方法**：
1. `web_search` / `image_search` 找官方或可信媒体图片 URL，`web_fetch` 确认来源可信。
2. `bash`（curl/wget）下载到工作目录 `assets/`，HTML 用本地相对路径或 base64 内联引用。
3. 每张图在图注标注来源（如「图：影石官方」）。只用产品本体/发布会现场/官方渲染图，不抓水印第三方作品、明星肖像、受版权艺术作品。

> **降级规则**：若下载失败（如沙箱网络白名单不含图片 CDN）或确实拿不到官方图，**用依官方规格手绘的 SVG 示意图替代**，并在图注**明确标注「示意图，非官方照片」**+给出官网链接。绝不用来源不明或可能侵权的图片，也不要硬凑。

## 第三步：固定报告结构（8 段，严格遵循）

**按以下 8 段顺序生成，每段都要有实质信息。** 参数维度按品类自适应（见末尾「品类参数预设」）。

### ① 产品概览 Hero（紧凑信息网格，不要大片留白）

把以下信息塞进一个**紧凑的信息网格卡片**（不是占满整屏的居中大标题）：
- **一句话总结**（这是什么、最大看点，一句话讲完）
- **发布日期** · **定位** · **起售价（¥）** · **目标用户**
- **★★★★★ 核心亮点评分**（主观，3–5 个维度各给星级，如 影像★★★★★ / 续航★★★☆☆）

右侧或下方可放一张产品主视觉图（官方图或 SVG 示意图）。Hero 整体控制在 1 屏内看完，信息要满。

### ② 🚀 本代升级 Top 5

最重要的 5 个升级点，编号 ①–⑤，每个一行标题 + 一句说明 + 一个标签（`首发` / `升级` / `同代领先`）。用紧凑列表或 2 列网格，不要每个升级占一大屏。

### ③ 📊 参数变化 Diff（核心，最花心思）

**这是 v4 的灵魂段。** 做一张 diff 表，逐维度对比「本机 vs 上代/对标」，并用箭头标变化方向：

| 维度 | 上代/对标 | 本机 | 变化 |
|------|-----------|------|------|
| SOC | … | … | ↑ 一句影响 |
| 影像 | … | … | ↑ … |
| 电池 | … | … | → 持平 |
| 存储 | … | … | ↓ 缩水（争议） |

- **箭头**：`↑` 升级 / `↓` 缩水或退步 / `→` 持平。用低饱和语义色（升级绿 #2e7d52、缩水赭 #b4623a、持平灰 #86868b）。
- **每行「变化」列都带一句**：这个变化是「真影响体验」还是「参数通胀」。这一句比数字本身更有价值。
- 全部维度都要呈现，**变化的行视觉上要更突出**（加粗箭头/浅底色）。

### ④ 🧠 AI 自动总结（4 个判断）

用 4 个小卡片回答 4 个问题（这是帮用户省时间的"结论先行"）：
- **为什么升级？**（厂商动机、市场背景）
- **✅ 真正影响体验的升级**（挑出 2–3 个用户能感知的）
- **⚠️ 只是规格变化 / 参数通胀**（哪些升级日常用不出差别）
- **📣 值得宣传的卖点**（营销层面的记忆点）

客观陈述，不替用户下"买不买"的结论。

### ⑤ 📈 与上代变化（可视化）

- 用 **↑↓→ 三类**归纳相比上代的提升 / 退步 / 持平项，附**变化原因**。
- 配一张 Chart.js 图：关键指标的「提升幅度 %」条形图，或「本代 vs 上代」分组条形图。
- 若为品类首作（无上代），本段改为「vs 同品类标杆」，并注明。

### ⑥ 🏆 竞品 SWOT

对比 1–2 个主要竞品，做**优势 / 不足 / 机会 / 风险** 四象限（2×2 网格卡片）：
- **优势**：本机比竞品强在哪
- **不足**：本机比竞品弱在哪
- **机会**：市场/生态层面的有利点
- **风险**：可能翻车/被反超的点

如实呈现争议，不偏袒。

### ⑦ 📋 全量参数（可折叠）

完整规格表，用 `<details><summary>展开全部规格</summary>…</details>` 折叠收纳，默认收起，避免占版面。按品类列全：SOC/RAM/ROM/屏幕/影像/电池/充电/接口/防护/尺寸重量/系统…

### ⑧ 📎 官方来源

底部小字列出所有数据与图片来源链接：官方规格页、官网产品页、引用的媒体评测。图片若用了 SVG 示意图，在此说明。

> 8 段是固定骨架，但每段**内部**可按产品类型灵活调整维度。顺序不要变。

## 第四步：生成 HTML 报告（密集版设计系统）

### 设计系统：克制但密集

**核心：信息满、对齐齐、扫读快。保留防丑底线，去掉"奢侈留白"。**

```css
/* 色彩 */
--bg:#ffffff;
--card-bg:#f5f5f7;
--text:#1d1d1f;
--text-secondary:#86868b;
--border:#d2d2d7;
--line:#e8e8ea;
--accent:#0071e3;
--brand:#xxxxxx;          /* 取产品标志色，低饱和，作点缀 */
/* 语义色（仅用于 diff 箭头/变化标记，低饱和） */
--up:#2e7d52;             /* 升级 */
--down:#b4623a;          /* 缩水/退步 */
--flat:#86868b;          /* 持平 */

/* 字体 */
font-family:-apple-system,'SF Pro Display','Helvetica Neue','PingFang SC',sans-serif;
数字一律 font-variant-numeric: tabular-nums;

/* 标题层级（比旧版收紧） */
h1:36px / 700 / -0.02em
h2:24px / 600          /* 段标题，带 emoji 前缀 */
h3:17px / 600
body:15px / 1.55       /* 正文比旧版小一档，更密 */
caption:12px / secondary

/* 间距（大幅收紧——这是解决"留白太多"的关键） */
段落间距（section padding）：44px  ← 旧版是 72–80px
卡片内边距：20px                  ← 旧版 32px
卡片圆角：14px
卡片用 --card-bg 背景区分，无边框

/* 布局：积极用多列 */
max-width:1040px        ← 略宽于旧版，容纳 diff 表与网格
能用 2 列就不用 1 列；Hero 信息网格、Top5、SWOT、AI 总结都用网格
窄屏（<768px）堆叠为单列
```

**保留的防丑底线（这些仍然禁止）：**
深色/暗黑背景、渐变色背景、霓虹/高饱和配色、发光效果、过重阴影、看起来像"AI 仪表盘"的浮夸元素。

**允许并鼓励（v4 新）：**
- **段标题用 emoji 前缀**（🚀📊🧠📈🏆📋📎）——这是结构锚点，帮快速定位。
- **diff 箭头的低饱和语义色**——这是信息，不是装饰。
- **紧凑的多列信息网格**——密度的主要来源。
- **可折叠 `<details>`**——收纳全量参数。

### 表格风格（加密版）

```css
table{width:100%;border-collapse:collapse;font-size:14px;}      /* 比旧版 15→14 */
th{text-align:left;padding:10px 14px;border-bottom:2px solid var(--text);
   font-weight:600;font-size:12px;color:var(--text-secondary);}
td{padding:10px 14px;border-bottom:1px solid var(--line);}      /* padding 14→10，更密 */
/* 无竖线、无外边框；diff 变化行可加浅底 background:#fafafa */
.up{color:var(--up);font-weight:600;}
.down{color:var(--down);font-weight:600;}
.flat{color:var(--flat);}
```

### 图片排版

```css
.hero-img,.media-img{width:100%;border-radius:14px;display:block;}
.img-caption{font-size:12px;color:var(--text-secondary);margin-top:6px;text-align:center;}
.color-gallery{display:grid;grid-template-columns:repeat(auto-fit,minmax(140px,1fr));gap:14px;}
```
Hero 配主视觉图；核心升级尽量配图；配色用 gallery。每图带 `.img-caption` 注来源。拿不到官方图时用 SVG 示意图并标注（见第二步降级规则）。

### 数据可视化：Chart.js

```html
<script src="https://cdn.jsdelivr.net/npm/chart.js@4"></script>
```
```javascript
Chart.defaults.color='#86868b';
Chart.defaults.borderColor='#e5e5e5';
Chart.defaults.font.family="-apple-system,'SF Pro Display','PingFang SC',sans-serif";
```
常用：**与上代提升幅度条形图**、**卖点雷达图**、**本代 vs 上代/竞品分组条形图**。控制在 2–4 张，每张有信息量；图表配色克制低饱和。

## 第五步：输出与预览

1. 下载的官方图放进 `assets/`（或 base64 内联）；拿不到则用标注好的 SVG 示意图。
2. HTML 写入文件，复制到输出目录。
3. 用 `present_files` 交付给用户。
4. 自检：8 段是否齐全、diff 箭头是否都带"影响判断"一句、价格是否全 ¥、图是否都有来源图注。
5. 问用户是否需要调整。

## 品类参数预设（第三步 Diff / 全量参数按此选维度）

不同品类的 Diff 表与全量参数，选对应维度，不要套手机维度去写相机：

- **手机**：SOC / GPU / 内存 RAM / 存储 ROM / 屏幕（尺寸·刷新率·峰值亮度）/ 影像（主摄·长焦·超广）/ 电池 / 充电 / AI 功能 / 系统 / 防护
- **芯片 / SoC**：制程 / CPU 架构与核心 / GPU / NPU 算力（TOPS）/ 能效 / 内存带宽 / 封装
- **云台 / 相机**：传感器尺寸 / 镜头与光圈 / 焦段·变焦 / 防抖 / 视频规格（分辨率·帧率·色深·log）/ 音频 / 屏幕 / 续航 / 存储 / 重量 / 防护
- **智能穿戴**：芯片 / 屏幕 / 健康传感器 / 电池续航 / 运动模式 / 防水 / 连接 / 系统
- **平板 / 笔电**：SOC / 内存 / 存储 / 屏幕 / 电池 / 接口 / 重量 / 系统 / 配件生态

## 注意事项

- **密度优先**：解决旧版"留白太多有效信息太少"是第一目标。每屏都要有可读取的数据/变化/判断。
- **变化是灵魂**：Diff 段每行变化都带"真影响还是参数通胀"一句；这比罗列规格值钱。
- **数据优先**：真实发布数据为准，没有的标「待官方公布」，绝不编造。官方与爆料冲突以官方为准。
- **价格统一人民币**：全篇 ¥，外币换算并注明原价。
- **图片真实可信**：官网/可信媒体官方素材，每图注来源；拿不到用标注清楚的 SVG 示意图，绝不用来源不明图。
- **克制判断**：客观陈述，争议如实呈现，不替用户下"买不买"结论。AI 总结段给判断，但不给购买指令。
- **单品为纲**：以这款产品纵深为主线，"与上代变化"和"竞品 SWOT"是两条对照轴，用来凸显它的价值，不要喧宾夺主变成横向评测。
- **响应式**：max-width + 媒体查询；多列网格在窄屏堆叠为单列。
