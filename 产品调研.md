# 发布会多品分页报告生成器 v1.2

针对**某一天内同场发布的多款新品**，生成**带分页 Tab 导航的高密度 HTML 报告**。

> v1.2 核心变更：精简/完整版开关改为**品类自动驱动**——手机/旗舰系列询问用户选择，其余生态品（耳机/手表/平板/芯片/配件等）固定精简版直接生成，不弹选项、不询问。同时继承 product-research v4.2 的 tooltip 修复与竞品选取规则。

## 核心设计理念

**全场速读 + 旗舰可深入 + 生态品快扫 + 联动可见。**

- **Tab 0 总览**：30 秒摸清「今天发了什么、最大看点、价格、各品怎么协同」。
- **Tab N 单品页**：品类驱动自动选版本深度，见下方「版本深度规则」。
- **生态联动矩阵**：各品协同功能，是发布会报告区别于单品报告的独有价值。
- **评分透明 + tooltip 修复**：全继承 product-research v4.2。

## ★ 版本深度规则（品类驱动，无需逐品询问）

| 品类 | 默认版本 | 是否询问用户 |
|---|---|---|
| **手机**（含折叠屏、旗舰系列、Ultra/Pro 旗舰） | **询问**：精简版还是完整版？ | ✅ 是，生成前问一次 |
| **耳机** | 精简版 | ❌ 否 |
| **手表 / 手环** | 精简版 | ❌ 否 |
| **平板** | 精简版 | ❌ 否 |
| **芯片 / SoC** | 精简版 | ❌ 否 |
| **路由 / 智能家居 / 配件** | 精简版 | ❌ 否 |
| **SUV / 汽车**（发布会预告型） | 精简版 | ❌ 否 |

**执行方式**：
- 识别品单后，若存在手机品类，**只问一次**：「[手机型号] 需要完整版（8段深度）还是精简版（4段快读）？其余品类已自动设为精简版。」
- 若品单里没有手机，直接全精简版生成，不询问任何选项。
- 用户回答后立即生成，不再追问。

### 精简版（4 段，所有非手机品类默认）

```
A: Hero（紧凑信息网格 + 主视觉 + 星级 tooltip）
B: 参数 Diff（▲▼＝ delta chip + 影响判断）
C: AI 自动总结（4 判断卡片）
D: SWOT（2×2）+ <details> 折叠全量参数
```

### 完整版（8 段，手机品类用户选择后）

```
① Hero  ② 🚀 Top5 升级（含 badge tooltip）
③ 📊 参数 Diff  ④ 🧠 AI 自动总结
⑤ 📈 与上代变化（Chart.js + 常驻依据小表）
⑥ 🏆 竞品 SWOT  ⑦ 📋 全量参数折叠  ⑧ 📎 来源
```

完整版 8 段的详细规范与 product-research v4.2 完全一致，参照该 skill 执行，此处不重复。

## 竞品选取原则（继承 v4.2）★

**以报告生成时间为基准，web_search 交叉验证真实竞品，不以调研产品发售时间截止。**

- 搜「[产品] 竞品 [当前年份]」「[产品] vs [对手]」，取媒体评测/用户社区明确提及的对标对象。
- 选**报告生成时已上市/预售的最新一代竞品**（消费者当前货架能买到的）。
- 若竞品有多代，优先最新代，除非旧代与调研产品价位更接近。Diff 表头注明「对标：[竞品]（[年份]，¥[价格]）」。
- 不以调研产品发布时间为截止线，避免选到已被替代的旧款误导读者。

## 执行流程

```
1. 明确场次    → 确认品牌/日期
2. ★ 品单确认  → web_search 搜出完整新品清单，展示给用户，等用户确认纳入哪些品
3. 版本确认    → 若含手机，询问一次精简/完整版
4. 数据收集    → 每品 web_search 规格 + 交叉验证竞品；搜生态联动功能
5. 规划分页    → Tab 0 总览 + Tab 1..N 各品（N ≤ 7）
6. 生成报告    → 单文件 HTML，固定 Tab 导航，原生 JS 切换
7. 自检交付    → present_files；告知各 Tab 版本深度；询问调整
```

## 第一步：明确场次 + 品单确认（★ v1.3 新增，必须执行）

**这是防止漏品/擅自取舍的关键步骤，不可跳过。**

1. 确认品牌与发布会名称、发布日期（用户已说明的直接用，不重复问）。
2. **web_search 搜该场发布会的完整新品清单**：
   ```
   "[品牌] [发布会名称或日期] 发布 新品 全部 汇总 [年份]"
   ```
3. 把搜索结果整理成品单表格，展示给用户，格式如下：

   | 序号 | 型号 | 品类 | 起售价（¥） | 备注 |
   |---|---|---|---|---|
   | 1 | 玄戒 O1 | 芯片 | 随整机 | |
   | 2 | 小米 15S Pro | 手机 | 3999 | |
   | … | … | … | … | … |

4. 询问用户：**「以上是本场发布会全部新品，请确认哪些纳入报告？可以全选，也可以指定某几个。」**
5. 等用户回复后，按用户指定的品单生成报告，不再自行增删。

> **为什么必须这样做**：Claude 可能因上下文干扰漏掉某个品（如之前漏掉了平板 7 Ultra），或擅自判断某品「不重要」而不写。品单确认把决策权交回用户，确保报告涵盖用户真正关心的产品。

**若含手机品类**：品单确认后，再追加一问：「[手机型号] 用精简版还是完整版？」（一次问完，其余品类自动精简版）

## 第二步：数据收集

```
每品（必做）：
  "[产品全名] 发布 规格参数 [年份]"
  "[产品全名] 竞品 对比 [当前年份]"   → 交叉验证竞品
  "[确认竞品] 规格 价格"               → 拿竞品参数喂 Diff

总览额外（必做）：
  "[品牌] 发布会 生态联动 协同 互联 [年份]"
```

## 第三步：Tab 结构规划

### Tab 0 — 发布会总览（固定，不可省略）

```
① 发布会概况行（品牌 · 日期 · 主题口号）
② 今日最大看点（3 条，每条一句话 + 影响判断）
③ 新品价格总表（所有品：型号 / 品类 / 起售价 ¥ / 一句话定位 / 版本深度标注）
④ 生态联动矩阵（各品协同点，只写有真实功能支撑的）
⑤ 本场核心数字（KPI 卡片 4 个）
```

### Tab N — 单品页

按版本深度规则执行。每品页底部加「← 上一品 / 下一品 →」跳转按钮。

## 第四步：HTML 实现规范

```javascript
function switchTab(n){
  document.querySelectorAll('.tab').forEach(b=>b.classList.remove('active'));
  document.querySelectorAll('.pane').forEach(p=>p.classList.remove('active'));
  document.querySelector('[data-tab="'+n+'"]').classList.add('active');
  document.getElementById('pane'+n).classList.add('active');
  window.scrollTo({top:0,behavior:'smooth'});
}
document.querySelectorAll('.tab').forEach(b=>
  b.addEventListener('click',()=>switchTab(+b.dataset.tab)));
```

## 第五步：自检与交付

**自检清单**：
- [ ] Tab 数 = 1（总览）+ 各品，导航齐全且切换正常
- [ ] 总览 5 部分（概况/看点/价格表含版本标注/生态联动/KPI）都有内容
- [ ] 手机品类已询问用户，其余品类直接精简版无需询问
- [ ] 各品竞品已 web_search 交叉验证，Diff 表头注明竞品名+年份+价格
- [ ] 所有 `.tip` 含 `white-space:normal` + `word-break:break-all`
- [ ] Diff 用 `.delta` ▲▼＝，每行带影响判断
- [ ] 完整版手机的得分图配常驻依据小表
- [ ] 价格全 ¥，无 $/€
- [ ] 响应式：Tab 窄屏横向滚动，多列网格窄屏堆叠

`present_files` 交付，告知各 Tab 是精简/完整版，询问是否调整。

## 注意事项

- **品类驱动版本**：手机询问一次，其余全精简，不要每品都问。
- **生态联动真实**：只写有真实功能支撑的协同点，不编造假联动。
- **竞品以当下为准**：选报告生成时最新已上市/预售的真实竞品。
- **Tooltip 不溢出**：`.tip` 必须含 `white-space:normal` + `word-break:break-all`。
- **品数 ≤ 7**：超过 7 品建议拆两份报告，单报告过大体验差。
- **Chart.js 懒初始化**：隐藏 Tab 的图表切换时才初始化，防 canvas 渲染错误。
